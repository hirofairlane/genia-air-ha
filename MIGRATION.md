# History Migration — genia-air-ha

> **Goal: zero data loss.** If you've been running raw `ebusd` MQTT discovery (or `genia-air-pack`) for months or years, none of that history should disappear when you install this integration.

There are **three storage layers** to think about. The integration handles the first two automatically. The third (InfluxDB) requires running a one-time script.

## Storage layer 1 — Home Assistant `entity_registry` (metadata)

What it stores: entity ↔ unique_id mapping, custom friendly_name, area, hidden_by, disabled_by, …

**Handled automatically** by the integration during setup. See `migration.py:async_migrate_legacy_unique_ids()`.

How: for each entity the integration is about to create, it looks up the **legacy unique_id** (from the previous LukasGrebe ebusd MQTT discovery) in the registry. If found, it transfers ownership to our integration's `config_entry` and updates the `unique_id` to the new stable name. HA's registry transition logic preserves `entity_id`, `name` (custom), `area_id`, etc.

You see no change — the entity stays where it was, with the name you gave it, in the dashboards you built.

## Storage layer 2 — HA Recorder (short-term states & statistics SQLite)

What it stores: state changes (`states` table), long-term statistics (`statistics`), keyed by `entity_id`.

**Handled automatically by Home Assistant itself.** Because layer 1's migration keeps the `entity_id` constant, Recorder doesn't need to know anything happened. Your history graphs and the Energy dashboard keep showing the full timeline.

Edge case: if you **decide to rename** the entity_id to the new English convention (`sensor.aerotermia_exterior_currentyieldpower` → `sensor.heat_pump_power_output`), HA's built-in rename flow (Settings → Devices → Entity → Rename) **does** migrate the Recorder statistics behind the scenes (since 2023.4). State history before the rename remains queryable via the old entity_id.

## Storage layer 3 — InfluxDB (long-term time series) ⚠️ REQUIRES SCRIPT

What it stores: time series in measurements with `entity_id` tag, keyed by domain (sensor/binary_sensor/…) and unit_of_measurement.

InfluxDB **does not follow** HA's entity rename. The data written under `sensor.aerotermia_exterior_currentconsumedpower` stays there forever, and new data starts being written under `sensor.heat_pump_power_input`. **Your Grafana dashboards break.**

That's what `scripts/migrate_influxdb.py` solves.

### What the script does

1. Reads a **mapping file** (`mapping.yaml`) with `old_entity_id → new_entity_id` pairs (the integration generates a recommended one for your install).
2. For each pair:
   - Queries all points in InfluxDB under the old entity_id (across all measurements/retention policies)
   - Re-writes them with the new entity_id tag
   - **Verifies count match** before deleting the old series (optional, opt-in)
3. Is **idempotent** (re-runs are safe) and supports **`--dry-run`** to preview.

### Running it

**Prerequisites**:
- Python 3.10+ on the machine running InfluxDB (or any machine with network access to it)
- `pip install influxdb` (for InfluxDB v1.x) or `pip install influxdb-client` (for InfluxDB v2.x)
- An InfluxDB admin token / username with read+write on the HA database

**Before you run anything**:
```bash
# Take a backup of your InfluxDB (essential)
influxd backup -portable /path/to/backup/  # v1.x
influx backup /path/to/backup/             # v2.x
```

**Dry run first**:
```bash
python scripts/migrate_influxdb.py \
  --influx-host 192.168.1.131 \
  --influx-port 8086 \
  --influx-db home_assistant \
  --influx-user homeassistant \
  --influx-password "$INFLUX_PASSWORD" \
  --mapping mapping.yaml \
  --dry-run
```

Reports:
```
Would migrate sensor.aerotermia_exterior_currentconsumedpower → sensor.heat_pump_power_input
  measurement: kW
  points found: 142,815
  date range: 2024-01-15 → 2026-05-23
  estimated time: ~4 min
Would migrate sensor.ebusd_ctls2_hc1heatcurve → sensor.heat_pump_heat_curve
  measurement: state
  points found: 9,234
  date range: 2024-03-02 → 2026-05-23
  estimated time: ~12 s
...
TOTAL: 38 entities, 2.1M points, ~25 minutes
```

**Actual migration**:
```bash
python scripts/migrate_influxdb.py \
  --influx-host 192.168.1.131 \
  --mapping mapping.yaml
  # WITHOUT --dry-run
```

By default the script **copies, does not delete**. If you want to remove old series after verification:
```bash
python scripts/migrate_influxdb.py --mapping mapping.yaml --delete-after-verify
```

### Mapping file format

```yaml
# mapping.yaml — generated by the integration, edit before running

# Power sensors
sensor.aerotermia_exterior_currentconsumedpower:  sensor.heat_pump_power_input
sensor.aerotermia_exterior_currentyieldpower:     sensor.heat_pump_power_output
sensor.ebusd_hmu_currentcompressorutil:           sensor.heat_pump_compressor_modulation
sensor.ebusd_hmu_waterthroughput:                 sensor.heat_pump_water_throughput

# Temperatures
sensor.aerotermia_exterior_status01_temp1:        sensor.heat_pump_flow_temp_supply
sensor.aerotermia_exterior_status01_temp1_2:      sensor.heat_pump_flow_temp_return
sensor.ebusd_ctls2_outsidetempavg_tempv:          sensor.heat_pump_outside_temp_avg

# Setpoints / config
number.ebusd_ctls2_z1manualtemp_tempv:            number.heat_pump_setpoint_manual
number.ebusd_ctls2_z1coolingtemp_tempv:           number.heat_pump_setpoint_cooling
sensor.ebusd_ctls2_hc1heatcurve:                  number.heat_pump_heat_curve

# Hours
sensor.aerotermia_exterior_hours_energy:          sensor.heat_pump_hours_total
sensor.aerotermia_exterior_hourshc_energy:        sensor.heat_pump_hours_heating
sensor.aerotermia_exterior_hourscool_energy:      sensor.heat_pump_hours_cooling

# Derived (from genia-air-pack v0.1 or the integration itself — if entity_id already
# matches the new naming, no migration needed; just delete the line)
# sensor.heat_pump_cop_instantaneous: sensor.heat_pump_cop_instantaneous
```

Lines where old=new are no-ops and can be removed.

### Edge cases handled

- **Multiple measurements per entity**: many entities have both a `state` and a `unit_of_measurement`-tagged measurement (e.g. `kW`). The script migrates all measurements for the old entity_id tag, not just one.
- **Multiple retention policies**: if you have downsampled data (CQ-generated), the script walks all retention policies.
- **Naming collisions**: if the new entity_id already has data (e.g. you re-installed and the integration already wrote some points), the script appends rather than overwrites, and warns.
- **InfluxDB v1 vs v2**: detected automatically from the connection parameters (`--influx-token` triggers v2 mode; `--influx-user/password` triggers v1 mode).
- **Big migrations**: paginated reads/writes with configurable `--batch-size` (default 10k points) — won't OOM the InfluxDB process.

### What the script does NOT do

- It does not delete old series unless you pass `--delete-after-verify` explicitly.
- It does not migrate Grafana dashboards (those reference entity_ids hardcoded — you have to find/replace yourself).
- It does not modify Home Assistant in any way (read-only on the HA side, write on the Influx side).

## Rollback

The integration provides a **rollback service**:

```yaml
service: genia_air.rollback_migration
```

That:
- Restores the legacy unique_ids on the entity_registry from a snapshot taken at install time
- Re-publishes the legacy MQTT discovery configs
- Disables (does not delete) the integration's own entities

After rollback, you're back to the LukasGrebe raw ebusd discovery as if nothing happened. Recorder history was always intact. InfluxDB history is now in *both* old and new entity_ids if you ran the migration — you can drop the new ones manually.

## Snapshot before installing

The integration takes a `core.entity_registry` snapshot to `/config/.genia_air/registry_backup_<timestamp>.json` before any registry mutation. Restorable via the rollback service.
